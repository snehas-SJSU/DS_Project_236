1. Project Overview
This document defines the complete API contract for our LinkedIn-like distributed system built for Data
236 — Distributed Systems The system is a three-tier, service-oriented application that combines
core professional networking features with an Agentic AI layer inspired by LinkedIn’s multi-agent
recruiting platform.
Rather than a monolithic backend, each feature area lives in its own microservice. Services talk to each
other through Kafka topics for asynchronous workflows, and expose REST endpoints for synchronous
requests from the client. An AI layer built in FastAPI adds recruiter-copilot capabilities on top.

10.1 Multi-Step Transaction Policy
For any operation that writes to the database AND publishes to Kafka (e.g., application submit), the
service uses an outbox pattern:
•
Write the record to MySQL inside a transaction.
•
Write the event to an outbox table inside the same transaction.
•
Commit the transaction.
•
A background relay process reads from the outbox and publishes to Kafka, retrying up to 5
times with exponential backoff.
•
If the relay exhausts retries, the event is moved to a dead-letter queue and an alert is raised.
The DB record remains; the system is consistent.
10.2 Idempotency Policy
Clients may include an Idempotency-Key: <uuid> header. Services store processed keys in Redis with
a 24-hour TTL. A duplicate request within that window returns the original response without
re-executing the operation.
Kafka consumers check the idempotency_
key from the event envelope before any DB write. If the key
already exists in the processed-events store, the consumer skips the write and commits the offset.

12. Agentic AI Evaluation Metrics
Per the project requirements, we report at least two measurable metrics for the AI components.
12.1 Matching Quality
We evaluate the job-candidate matching skill using a manual rubric on a sample of 50 shortlists. For
each shortlist we ask: do the top-5 candidates have at least 70% skills overlap with the job? We report
a top-5 precision score.
•
Metric: Top-5 Skills Overlap Precision (target ≥ 80%)
•
Method: manual review by two group members on 50 randomly sampled job-shortlist pairs.
•
Baseline: keyword-only matching (no embeddings) for comparison.
12.2 Human-in-the-Loop Approval Rate
We track recruiter decisions on AI-generated outreach drafts over the course of the demo dataset:
•
% approved as-is (draft was good enough to send without edits)
•
% edited before sending
•
% rejected outright
Target: ≥60% approved or edited, indicating the AI output is useful to recruiters rather than noise.
13. Performance & Scalability Notes
The API contract is designed to support the benchmark scenarios required by the project. Below we
document the expected behavior under load and the caching/Kafka configuration that supports it.
13.1 Benchmark Scenario A — Job Search + Detail View
•
Endpoint: POST /jobs/search followed by POST /jobs/get.
•
Caching: Job detail responses are cached in Redis (TTL 5 minutes). Under 100 concurrent
threads the cache hit rate should be >90% for popular postings.
•
Expected: <200 ms p95 latency at 100 concurrent users (B+S configuration).
13.2 Benchmark Scenario B — Application Submit
•
Endpoint: POST /applications/submit.
•
This is a write-heavy path: MySQL insert + Kafka publish + Redis cache invalidation.
•
Expected: <500 ms p95 latency at 100 concurrent users (B+S+K configuration).
•
Kafka decouples the analytics pipeline from the synchronous path; analytics lag up to 1 second
is acceptable.

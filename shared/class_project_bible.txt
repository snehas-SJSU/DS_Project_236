1. Project Overview
LinkedIn is a professional networking and hiring platform. In this project, your team will build a LinkedIn-like system using a distributed, service-oriented architecture. In addition to core LinkedIn features (profiles, job postings, applications, connections, and messaging), you will implement an Agentic AI layer inspired by LinkedIn’s multi-agent platform approach: agents exposed as services, coordinated through messaging for multi-step workflows.

6.1 Kafka requirements
	•	Define Kafka topics per domain event (e.g., job.viewed, job.saved, application.submitted, message.sent, connection.requested).
	•	Use consumer groups for scalability and fault tolerance.
	•	At least one end-to-end workflow must be asynchronous (UI produces → Kafka → backend consumes → DB update → UI notification).
Kafka event payload standard (Required)
All Kafka messages must follow a shared JSON envelope:
{
  "event_type": "job.viewed | job.saved | application.submitted | message.sent | connection.requested | ai.requested | ai.completed",
  "trace_id": "uuid",
  "timestamp": "ISO-8601",
  "actor_id": "member_id or recruiter_id",
  "entity": { "entity_type": "job|application|thread|connection|ai_task", "entity_id": "..." },
  "payload": { "domain_specific_fields": "..." },
  "idempotency_key": "uuid-or-hash"
}  
Rules
	•	event_type must match the topic naming convention (or be derivable from it).
	•	trace_id must stay the same across a multi-step workflow (UI → Kafka → services → AI).
	•	idempotency_key must be used by consumers to avoid duplicate DB writes on retries.
	•	Consumers must handle at-least-once delivery safely (idempotent processing).
Example
	•	Topic: application.submitted
	•	event_type: application.submitted
	•	payload contains: job_id, member_id, resume_ref, submission_metadata 
7. Agentic AI Services 
Build an Agentic AI layer inspired by LinkedIn’s multi-agent platform: agents are services with reusable skills, and multi-step workflows are coordinated via messaging. Agents should use human-in-the-loop checkpoints for safety when generating recruiter-facing actions (e.g., outreach messages).  Agentic AI Recommendation Service in FastAPI
It behaves less like a search box and more like a recruiter copilot that plans multi-step hiring workflows (shortlist → outreach draft → approval → send). Use FastAPI for HTTP + WebSockets to push updates to the UI. (FastAPI)
7.1 AI Services to Implement
Your team must implement the following AI skills/services from the list below. These services should be designed as stateless microservices exposed via REST or gRPC.
	•	Resume Parser Skill: extract structured fields (skills, years of experience, education) from resume text.
	•	Hiring Assistant Agent (Supervisor)
	•	Orchestrates other AI skills to shortlist candidates for a job posting and coordinate multi-step workflows.
	•	Job–Candidate Matching Skill
	•	Computes a match score using embeddings + rules (e.g., skills overlap, location, seniority).
	•	Career Coach Agent (optional but encouraged)
	•	Helps members tailor their resume/headline for a target job and provides suggestions with rationale.
7.2 Kafka-Based Multi-Agent Orchestration (Required) 
You must implement at least one AI workflow that is multi-step and Kafka-orchestrated, using the Supervisor + at least two skills.
Kafka topics
	•	Create an ai.requests topic for AI tasks.
	•	Create an ai.results topic for AI outputs/events.
Workflow requirements
	•	Each AI skill runs as a stateless service (REST/stream).
	•	The Hiring Assistant (Supervisor) consumes from ai.requests, calls the required skills/services, and emits results to ai.results.
	•	Implement at least one long-lived task as a sequence of messages (multi-step pipeline) with a shared trace_id for end-to-end observability.
